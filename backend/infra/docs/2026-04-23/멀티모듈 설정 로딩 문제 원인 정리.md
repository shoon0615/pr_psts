# 멀티모듈 설정 로딩 문제 원인 정리

## 배경

`api` 모듈에서 애플리케이션 실행 중 아래 오류가 발생했다.

```
Could not resolve placeholder 'spring.elasticsearch.uris'
in value "${spring.elasticsearch.uris}"
```

겉으로는 `SpringSecurityConfig` 생성자 주입 문제로 보였지만,
재검증 결과 실제 부팅 실패 원인은 `ElasticsearchConfig`의 프로퍼티 주입 실패였다.

---

## 실제 원인

- `ElasticsearchConfig`는 `infra` 모듈에 존재
- `@Value("${spring.elasticsearch.uris}")`로 설정값을 주입받음
- 설정값은 기존에 `infra/src/main/resources/application.yml`에만 존재
- 하지만 멀티모듈에서 `application.yml` 파일은 모듈별로 자동 병합을 보장하지 않으므로,
  실행 시점에 `api` 기준 classpath에서 해당 키를 안정적으로 찾지 못함

즉, **모듈 분리 자체가 문제라기보다 "설정 파일 로딩 경로를 명시하지 않은 것"이 원인**이었다.

---

## 조치 내용

### 1) infra 전용 설정 파일 분리

신규 파일 생성:

- `backend/infra/src/main/resources/infra-elasticsearch.yml`

내용:

```yaml
spring:
  elasticsearch:
    uris: http://elasticsearch:9200
  data:
    elasticsearch:
      repositories:
        enabled: true
```

### 2) api에서 명시적으로 import

수정 파일:

- `backend/api/src/main/resources/application.yml`

추가 내용:

```yaml
spring:
  config:
    import: optional:classpath:infra-elasticsearch.yml
```

### 3) 기존 infra application.yml 제거

- 삭제 파일: `backend/infra/src/main/resources/application.yml`
- 이유: 동일 파일명(`application.yml`) 중복으로 인한 로딩 혼선 제거

---

## 검증 결과

- `bash gradlew clean :api:bootRun`으로 재실행
- `spring.elasticsearch.uris` placeholder 오류 재발 없음
- 애플리케이션 정상 기동 확인

---

## 운영 가이드

- 공통/인프라 설정은 모듈별 전용 파일(`infra-*.yml`)로 분리
- 실행 모듈(`api`)에서 `spring.config.import`로 로딩 대상을 명시
- 환경별 차이는 가능하면 환경변수(`SPRING_ELASTICSEARCH_URIS`) 우선 사용

이 원칙을 따르면 멀티모듈 구조를 유지하면서도 설정 로딩을 예측 가능하게 관리할 수 있다.
